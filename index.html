Section 1: Problems Encountered in the Map

After initially downloading the open street map of Irvine and running it against a provisional data.py file, I noticed that the data has abbreviated street names for examples: Edinger Ave, Dyer Rd West...etc

Abbreviated Street Names

Once the data was imported to MongoDB, some basic querying revealed street name abbreviations. I updated all substrings in problematic address strings, such that “Dyer Rd West” becomes “Dyer Road West”.

# Sort postcodes by count, descending
> db.project3.aggregate([{"$match":{"address.postcode":{"$exists":1}}}, {"$group":{"_id":"$address.postcode", "count":{"$sum":1}}}, {"$sort":{"count":­1}}])

Here are the top two results, beginning with the highest count:

[ {"_id" : "92630", "count" : 14815},
{"_id" : "92620", "count" : 7518}, ...

Considering the relatively few documents that included postal codes, of those, it appears that out of the top ten, two aren’t in Irvine. That struck me as surprisingly high to be a blatant error, and found that the number one postal code "92630" lies in Lake Forest, CA. So, I performed another aggregation to verify a certain suspicion...

# Sort cities by count, descending
> db.project3.aggregate([{"$match":{"address.city":{"$exists":1}}},{"$group":{"_id":"$address.city", "count":{"$sum":1}}}, {"$sort":{"count":­1}}])

And, the results, edited for readability:

[{ "_id":"Irvine", "count":25014 },
{ "_id":"Lake Forest", "count":14556 },
{ "_id":"Tustin", "count":2181 },...

These results confirmed my suspicion that this metro extract would perhaps be more aptly named “Orange County's Metropolitan Area” for its inclusion of surrounding cities in the sprawl. So, these postal codes aren’t “incorrect,” but simply unexpected.

Section 3: Additional Ideas

Contributor statistics and gamification suggestion The contributions of users seems incredibly skewed, possibly due to automated versus manual map editing (the word “bot” appears in some usernames). Here are some user percentage statistics: Top user contribution percentage (“SJFriedl”) - 50.60% Combined top 2 users' contribution (“SJFriedl” and “Aaron Lidman”) - 60.84% Combined Top 10 users contribution - 86.20% Combined number of users making up only 1.7% of posts (7303/422890) - 400 (about 77.5% of all users) Thinking about these user percentages, I’m reminded of “gamification” as a motivating force for contribution. In the context of the OpenStreetMap, if user data were more prominently displayed, perhaps others would take an initiative in submitting more edits to the map.

Section 4: Conclusion

After this review of the data it’s obvious that the Irvine city area is incomplete, though I believe it has been well cleaned for the purposes of this exercise. It interests me to notice a fair amount of GPS data makes it into OpenStreetMap.org on account of users’ efforts. With a rough GPS data processor in place and working together with a more robust data processor similar to data.pyI think it would be possible to input a great amount of cleaned data to OpenStreetMap.org.
